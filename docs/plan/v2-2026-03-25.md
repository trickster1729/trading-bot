## Overview

An algorithmic trading system targeting consistent returns across Stocks, Crypto, and Options.
Built in phases: starting as a local CLI with shadow/backtesting, graduating to a cloud-hosted
live trading engine. All layers are pluggable — strategies, brokers, data sources, and models
can be swapped without rewriting core logic.

---

## Goals

- Target: 5–10%+ monthly returns with controlled drawdown
- Starting capital (once live): $5,000–$10,000
- Risk first: 2% max per trade, 10% drawdown = hard halt
- Long-term: cloud-hosted, fully automated, multi-asset, multi-strategy
- Side goal (P1): co-author research paper with Akanksha for publication (arXiv/SSRN)

---

## Build Phases

### Phase 1 — CLI + Shadow Trading (in progress)
- Local Python CLI
- Historical data replay (Yahoo Finance for stocks, CCXT for crypto)
- Paper PnL tracking — no real money
- One baseline strategy per asset class (RSI + SMA momentum)
- Backtest harness with performance report (Sharpe, drawdown, win rate)
- Persistent DB: every trade, signal, and run stored in SQLite
- **Gate:** positive Sharpe, win rate > 50%, max drawdown < 15% over 2-year backtest

### Phase 2 — Paper Live Trading
- Connect to live market data feeds (real prices)
- Alpaca paper account (stocks/options) + Binance testnet (crypto)
- Same code as Phase 1 — swap DataLoader and Broker instances only
- Run continuously 4–8 weeks, log every decision
- Add: MeanReversion strategy, health monitoring, DB-backed `bot report`
- **Gate:** positive expectancy over 100+ live paper trades

### Phase 3 — Small Live Trading
- Flip Alpaca + Binance from paper to live accounts
- Start with $5,000–$10,000 real capital
- Logs → AWS CloudWatch, alerts → AWS SNS
- Add: LLM sentiment (Claude API), ML classifier signal
- **Gate:** profitable for 4+ consecutive weeks

### Phase 4 — Scale + Cloud
- Docker + AWS ECS Fargate
- Prometheus + Grafana dashboard (local Docker in Phase 3, cloud in Phase 4)
- Add IBKR for options, Coinbase Advanced as Binance fallback
- Multi-strategy portfolio (uncorrelated signals in parallel)
- RL agent (PPO/SAC) for execution timing

## Architecture

```
Data Ingestion Layer
  ├── Yahoo Finance (stocks, free historical)
  ├── Alpaca WebSocket (stocks, live — Phase 2+)
  └── CCXT (crypto, unified API)

Signal Engine
  ├── Strategy ABC → generate_signals(bars, symbol) → list[Signal]
  └── Implementations: Momentum (Phase 1), MeanReversion (Phase 2),
                       LLMSentiment (Phase 3), MLClassifier (Phase 3), RLAgent (Phase 4)

Risk Management
  ├── Fixed fractional sizing (2% default, configurable)
  ├── Per-asset-class limit overrides (tighter for crypto/options)
  ├── Max drawdown guard (10% → kill switch)
  └── Max open positions, per-symbol limits

Execution Layer
  ├── Broker ABC → submit_order(signal, qty) → OrderResult
  └── Implementations: PaperBroker (Phase 1), AlpacaBroker (Phase 2+), BinanceBroker (Phase 2+)

Persistence
  ├── TradeRecord — every closed trade with full signal context
  ├── SignalRecord — every signal (approved + blocked)
  └── BacktestRun — run metadata + performance summary

Monitoring
  ├── structlog — structured JSON logs (file) + Rich console
  ├── PerformanceTracker — live PnL, Sharpe, drawdown (in-memory)
  ├── Health checks — heartbeat loop (Phase 2+)
  └── Alerts — console (Phase 1) → AWS SNS (Phase 3+)
```

**When to add an ADR:**
- Choosing between two reasonable alternatives (language, framework, library, algorithm)
- A decision that will be expensive to reverse later
- Any choice that a future contributor would reasonably ask "why did they do it this way?"

## Key Architectural Decisions (summary — see docs/adr/ for full rationale)

| Decision | Choice | Key reason |
|---|---|---|
| Language | Python 3.11+ | Ecosystem depth for quant/ML |
| Architecture | Pluggable ABCs | Testability, extensibility, phase parity |
| Logging | structlog | Structured JSON + human-readable console |
| Build phases | Shadow → Paper → Live | Derisk capital deployment |
| Position sizing | Fixed fractional (2%) | Bounded loss, no parameter estimation needed |
| Database | SQLAlchemy + SQLite → Postgres | Zero config now, one URL to scale |
| Brokers | Alpaca + Binance | Free paper trading, best APIs |
| Phase 1 strategy | RSI + SMA momentum | Interpretable, well-studied, pipeline-proving |

---

## Risk Rules (non-negotiable from day 1)

- Max 2% of account per trade
- Max 10% drawdown → halt all trading, alert
- No leverage in Phase 1–2
- No overnight positions in Phase 1 (equities)
- All trades logged with full context (signal, confidence, entry, exit, PnL)
- Kill switch: `bot stop --force` halts all open orders immediately
